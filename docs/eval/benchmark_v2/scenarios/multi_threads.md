1. 쓰레드 로컬 오염 (Thread-Local Leakage)
시나리오: 쓰레드풀은 쓰레드를 재사용합니다. 이전 작업이 threading.local()에 저장한 사용자 정보(예: is_admin=True)를 지우지 않고 종료합니다.

분석 포인트: 다음 작업은 일반 유저인데, 재사용된 쓰레드 덕분에 관리자 권한을 갖게 됩니다.

왜 어려운가: 코드상으로는 권한 부여 로직이 완벽해 보입니다. 에이전트는 "이전 작업의 흔적"이라는 시계열적 맥락을 짚어내야 합니다.

1. 레이스 컨디션: "잃어버린 업데이트" (The Lost Update)
시나리오: 두 쓰레드가 공유 딕셔너리(self.cache)의 중첩된 값을 동시에 수정합니다. (예: cache['count']['val'] += 1)

분석 포인트: 파이썬의 GIL 덕분에 단일 연산은 안전해 보이지만, 중첩 구조를 읽고 쓰는 사이의 미세한 틈에서 업데이트가 누락됩니다.

왜 어려운가: 로그에는 두 쓰레드 모두 "업데이트 완료"라고 찍히지만, 최종 결과값이 기대값보다 작습니다. 수학적 계산과 쓰레드 간섭을 매칭해야 합니다.

1. 클로저 캡처 버그 (The Closure Trap in Loops)
시나리오: for 루프 안에서 쓰레드를 생성하며 lambda나 함수를 전달할 때, 루프 변수(i)를 인자로 넘기지 않고 참조만 합니다.

분석 포인트: 모든 쓰레드가 실행될 시점에는 루프가 이미 끝나 있어, 모든 쓰레드가 마지막 인덱스 값만 처리합니다.

왜 어려운가: 코드는 매우 깔끔해 보입니다. LLM이 "함수가 실행되는 시점의 변수 스코프"라는 언어적 특성을 이해해야 풀 수 있습니다.

1. 제너레이터 소진 (Generator Exhaustion)
시나리오: 대량의 데이터를 yield하는 제너레이터를 여러 쓰레드에 공유해서 넘깁니다.

분석 포인트: 한 쓰레드가 데이터를 다 써버리면(Exhaust), 다른 쓰레드는 빈 데이터를 받게 됩니다.

왜 어려운가: 데이터 소스 자체는 문제가 없는데, "한 번 읽으면 사라지는" 제너레이터의 특성을 간과하면 왜 데이터가 None인지 찾기 힘듭니다.

1. 가변 인자 디폴트의 덫 (Mutable Default Argument)
시나리오: def add_task(tasks=[]) 처럼 리스트를 기본 인자로 사용합니다.

분석 포인트: 이 리스트는 함수 정의 시점에 딱 한 번 생성되어 공유됩니다. 여러 쓰레드가 이 함수를 호출할수록 tasks 리스트는 의도치 않게 계속 비대해집니다.

왜 어려운가: 로컬 테스트 한두 번으론 안 터집니다. "함수 호출 횟수가 늘어날수록 메모리가 오염되는 과정"을 추론해야 합니다.

1. 부동 소수점 누적 오차 (Floating Point Drift)
시나리오: 미로 탐색이나 거리 계산에서 0.1을 10번 더합니다.

분석 포인트: 결과가 1.0이 아니라 0.999999999999가 됩니다. 이후 if value == 1.0: 체크에서 탈출에 실패합니다.

왜 어려운가: 로그에는 1.0으로 반올림되어 찍힐 수 있습니다. 로그와 실제 값 사이의 정밀도 차이를 의심해야 합니다.

1. 미래의 유령 (The Unawaited Future)
시나리오: executor.submit()으로 중요한 작업을 던졌는데, 예외가 발생합니다. 하지만 메인 쓰레드는 future.result()를 호출하지 않고 프로그램을 종료하려 합니다.

분석 포인트: 예외는 발생했지만 로그에도 안 남고 조용히 사라집니다(Swallowed). 프로그램은 그냥 "이유 없이" 결과가 안 나온 상태가 됩니다.

왜 어려운가: "에러가 발생하지 않은 것이 에러"인 상황입니다. 실행되지 않은 코드 블록을 찾아내야 합니다.

1. 순환 참조와 가비지 컬렉션 (WeakRef GC Issue)
시나리오: 메모리 관리를 위해 weakref를 썼는데, 쓰레드가 작업을 시작하기도 전에 메인 쓰레드에서 객체 참조가 사라져 GC(가비지 컬렉션)가 객체를 지워버립니다.

분석 포인트: 쓰레드가 막상 일을 하려고 보니 참조하던 객체가 None이 되어 있습니다.

왜 어려운가: 객체가 '살아있어야 하는 기간'에 대한 설계 미스입니다. 객체의 **생명주기(Lifecycle)**를 분석해야 합니다.

1. 우선순위 역전 (Priority Inversion - Logic level)
시나리오: 중요한 '긴급 탈출' 쓰레드가 낮은 우선순위의 '청소' 쓰레드가 잡고 있는 Lock을 기다리느라 영원히 실행되지 않습니다.

분석 포인트: 교착 상태(Deadlock)는 아니지만, 시스템이 극도로 느려지거나 특정 작업이 무시됩니다.

왜 어려운가: 코드는 동작합니다. 다만 성능 로그의 타임스탬프를 보고 "왜 이 작업이 여기서 5초나 걸렸지?"를 분석해야 합니다.

1. 순서 의존성 버그 (Non-Deterministic Order)
시나리오: 쓰레드 A는 '초기화', 쓰레드 B는 '실행'을 담당합니다. 보통 A가 빠르지만, 시스템 부하가 걸리면 B가 먼저 실행되어 초기화되지 않은 메모리에 접근합니다.

분석 포인트: 실행할 때마다 결과가 달라지는 '하이젠버그'입니다.

왜 어려운가: "우연히 성공하는 경우"가 많기 때문입니다. **인과관계(Happened-before relationship)**가 코드에 강제되어 있는지 검증해야 합니다.
