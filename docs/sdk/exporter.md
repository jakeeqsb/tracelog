# exporter.py — Design Document

## Role and Purpose

`exporter.py` is the **Pluggable Dump Target** layer for the TraceLog SDK.

It is responsible for deciding where and how to persist the Trace-DSL entries (`LogEntry` list) extracted from the `ChunkBuffer` when an `ERROR` occurs. In the MVP, the exporter does not render the final human-facing trace block. Instead, it serializes an assembly-friendly JSON dump that can later be stitched together by the Aggregator.

By isolating this responsibility from `handler.py`, the SDK can support file sinks, stream sinks, and future network-based exporters without modifying the core logging flow.

---

## Class Design

### `TraceExporter` (Protocol/ABC)

**What it is:** An abstract base class that all Exporter implementations must follow.

**Interface:**

```python
class TraceExporter(ABC):
    @abstractmethod
    def export(self, entries: List[LogEntry]) -> None:
        """Serialize and persist the extracted LogEntry list as a JSON dump."""
```

**Design Decision:**

- Passing the raw `LogEntry` list gives each exporter full access to timestamps, levels, and DSL lines while keeping serialization policy outside the handler.
- Exporters are treated as dump serializers, not as final trace assemblers.

### MVP Dump Schema

For the MVP, exporters emit at least the following JSON payload:

```json
{
  "trace_id": "TRACE123",
  "span_id": "SPANBBBB",
  "parent_span_id": "AAAA",
  "timestamp": "2026-03-08T10:00:00Z",
  "dsl_lines": [
    ">> send_email(user=\"jake@example.com\")",
    "!! ConnectionError: SMTP server unreachable"
  ]
}
```

`dsl_lines` are the original Trace-DSL lines already generated by the SDK. The Aggregator later uses the metadata fields to assemble multiple dumps into one unified trace.

`trace_id` is stable for the full request or task. `span_id` identifies the current active span within that trace.

### `StreamExporter`

**What it is:** The default exporter that writes one JSON dump to a writable stream (for example `sys.stderr` or `sys.stdout`).

**Features:**

- Emits one structured payload per error event, which downstream ingestion can parse without reparsing a human-oriented header/footer format.
- Fits container and server environments where stdout/stderr are already collected.
- Can still include timestamp information inside the JSON payload.

---

### `FileExporter`

**What it is:** An exporter that appends JSON dumps to a file on local disk.

**Features:**

- Useful for long-running services and file-based ingestion workers.
- **Supports Automatic Rotation (`max_bytes`)**: Preemptively prevents solitary files from ballooning endlessly. If the file crosses the size ceiling, it swaps to a `.bak` backup file and refreshes with an empty file instance.
- Safely initializes uncreated parent directories during its first `export()` run to prevent I/O errors minus permission lockouts, leaning into the Zero-Friction mantra.
- Produces a format that an Aggregator can scan incrementally and group by `trace_id`.

**Interface Example:**

```python
# Maintain up to 10MB; beyond this, overwrites onto trace.log.bak
exporter = FileExporter("/var/log/trace.log", max_bytes=10*1024*1024)
```

## Execution Flow (From TraceLogHandler's Perspective)

Inside `handler.py`, outputs are delegated to the configured exporter:

```python
# Inside tracelog/handler.py _dump method

def _dump(self, buf: ChunkBuffer) -> None:
    # 1. Snapshot the buffer and clear it atomically
    entries = buf.flash()

    # 2. Delegate JSON dump serialization and persistence
    self._exporter.export(entries)
```

This separation allows future exporters such as `NetworkExporter` or a dedicated Aggregator sink to be added without changing the SDK core.
